וכנית עבודה: Log-RAG Analyzer
אבן דרך 1: תשתית וגמישות (Foundations)
בשלב זה נבנה את "עמוד השדרה" של האפליקציה כך שתהיה ניידת וקלה להחלפת מודלים.

הגדרת סביבה: יצירת requirements.txt ושימוש ב-.env לניהול מפתחות API (בלי להכניס אותם לקוד).

בניית ה-Wrapper: יצירת מחלקת AIService שתפריד בין הלוגיקה של Gemini לשאר התוכנה.

בדיקה: הרצת סקריפט קטן שמוודא שיש תקשורת תקינה מול ה-API של Gemini ומקבלים תשובה ראשונית.

אבן דרך 2: שלב ה-Ingestion (Stage A)
הפיכת קובץ הלוג למידע וקטורי.

Log Loader: כתיבת קוד שקורא קבצי .log.

Smart Chunking: פיצול הטקסט לנתחים תוך שמירה על שלמות שורות הלוג.

Vector DB (ChromaDB): יצירת מסד הנתונים המקומי ושמירת ה-Embeddings בתוכו.

בדיקה: בדיקה פיזית של תיקיית ה-DB (לוודא שנוצרו קבצים) והדפסת כמות הנתחים (Chunks) שנוצרו.

אבן דרך 3: שלב ה-Retrieval (Stage B)
יכולת "למצוא" את המחט בערימת השחת.

Similarity Search: כתיבת פונקציה שמקבלת שאלה ומחזירה את 3-5 שורות הלוג הכי רלוונטיות.

בדיקה: נשאל שאלה כמו "חפש שגיאות בשרת" ונראה אם המערכת מחזירה שורות לוג שבאמת מכילות את המילה ERROR או קוד שגיאה. זהו שלב קריטי שבו בודקים את איכות ה-Embedding.

אבן דרך 4: שלב ה-Generation (Stage C)
חיבור הכל לתשובה אנושית.

Prompt Engineering: בניית התבנית (Template) שתנחה את Gemini לענות רק על סמך הלוגים.

RAG Chain: חיבור של השליפה (Stage B) עם המודל (Stage C).

בדיקה: הרצת שאילתה מלאה וקבלת הסבר טקסטואלי על תקלה שתועדה בלוג.

אבן דרך 5: הכללה וניידות (Final Polish)
Main Interface: יצירת ממשק פשוט (CLI או סקריפט ראשי) שמריץ את כל התהליך."

Dockerization (אופציונלי): הכנת Dockerfile כדי שתוכל להריץ את זה על כל מחשב בלחיצת כפתור.
